# Product Requirements Document - Content Repurposer（适文）

**Author:** dadong
**Date:** 2026-03-24
**Revision:** 2（经专家评审后修订）

## Executive Summary

适文是一款 Web 应用，为内容创作者提供"语义级"跨平台内容改写能力。用户输入一篇已有文章，系统理解其核心内容后，按目标平台（小红书、微信公众号、知乎）的内容规范、语言风格和互动模式重新组织表达，输出包含改写文案、备选标题、推荐标签和互动引导语的完整"内容发布包"。

目标用户覆盖个人自媒体博主、企业新媒体运营和知识付费从业者。产品 MVP 阶段以**免费优先**策略运营 -- 先用免费产品建立用户基础和口碑，验证改写质量和产品市场契合度，再引入付费模式。

系统定位为内容创作链条中的"翻译层" -- 不替代分发工具（融媒宝/蚁小二），不替代排版工具（壹伴/有一云），只做"已有内容的跨平台语义改写"这一个垂直环节。

### What Makes This Special

**核心差异化：语义级改写 vs 句子级替换。** 现有工具的"改写"本质是同义词替换或简单缩写/扩写，输出结果在目标平台上读起来仍像"搬运内容"。适文的改写从内容结构层面重组 -- 同一个核心内容，在小红书以"体验分享+结论前置"的方式表达，在公众号以"论点递进+深度分析"的方式表达，在知乎以"问题拆解+数据支撑"的方式表达。

**与通用 AI 的关键区别：开箱即用。** 用户使用 ChatGPT 做跨平台改写需要自己编写 prompt、理解各平台内容规则、逐平台调试输出质量。适文将这些专业知识内置为产品能力 -- 预置平台专属 prompt 链、真实爆文语料 few-shot 示例、实时更新的平台规则适配层。用户零学习成本，粘贴文章、选平台、复制结果，4 步完成。

**产品愿景：** 成为连接所有内容平台的"内容翻译层"基础设施。

## Project Classification

- **项目类型：** Web Application（MVP 阶段以 C 端个人用户为主）
- **领域：** 内容科技 / 新媒体工具
- **复杂度：** 中等 -- 核心 AI 改写逻辑有深度，但系统架构相对直接
- **项目背景：** 全新项目（Greenfield），无历史代码或遗留系统

## Success Criteria

### User Success

- **核心成功场景：** 用户粘贴一篇公众号文章，选择"小红书"，数秒内获得一篇读起来像原生小红书帖子的改写结果，无需大幅修改即可直接发布
- **"Aha" 时刻：** 用户第一次看到改写结果时发现"这比我自己改写的还好" -- 不只是语言风格对了，内容结构也完全符合目标平台的阅读习惯
- **情感目标：** 从"跨平台发内容好累"变成"跨平台发内容好简单"

### Business Success

MVP 阶段采用**免费优先**策略，先验证产品价值再探索商业化路径。

| 指标 | 3 个月目标（免费期） | 12 个月目标（含付费） |
|---|---|---|
| 注册用户 | 5,000+ | 50,000+ |
| 月活跃用户（MAU） | 2,000+ | 20,000+ |
| 日均改写次数 | 1,000+ | 10,000+ |
| 用户反馈满意率 | > 70%（"有帮助"反馈比例） | > 80% |
| 付费用户数 | 不适用（免费期） | 3,000+（转化率 6%） |
| MRR | 不适用 | 150,000+ 元 |

### Technical Success

| 指标 | 目标值 |
|---|---|
| 改写响应时间（首 token） | < 2 秒 |
| 改写完成时间（全文） | < 15 秒（2000 字以内原文） |
| 系统可用性 | 99.5%+ |
| API 错误率 | < 1% |
| 单次改写 API 成本 | < 0.3 元（免费期成本可控性是生存前提） |

### Measurable Outcomes

- **改写可用率：** 用户复制改写结果后，不返回重新生成的比例 > 70%（行为数据推断）
- **用户主动反馈：** 每次改写后"有帮助/没帮助"反馈中，正面反馈 > 70%（主动反馈数据）
- **留存率：** 7 日留存 > 30%，30 日留存 > 15%
- **NPS：** > 40（应用内定期调研）
- **平均每用户每日改写次数：** > 2 次（表明工具已融入工作流）

## Product Scope

### MVP - Minimum Viable Product（Phase 1）

**核心原则：免费优先，只做改写，做到极致。**

| 能力 | 说明 |
|---|---|
| 文章输入 | 粘贴全文为主，URL 提取为辅（Best Effort，失败时引导粘贴） |
| 3 个目标平台 | 小红书、微信公众号、知乎 |
| AI 改写引擎 | 识别内容类型后，按目标平台的内容结构和语言风格重组表达 |
| 内容发布包输出 | 改写文案 + 3 个备选标题 + 推荐标签 + 互动引导语 |
| 语气风格选择 | 三档预设（口语化 / 标准 / 正式），选择后重新生成 |
| 流式输出 | 改写结果逐字呈现，消除等待焦虑 |
| 改写反馈 | 每次改写后"有帮助/没帮助"反馈按钮 |
| 历史记录 | 保存改写历史，支持查看和复用 |
| 用户系统 | 注册/登录（手机号 + 微信 OAuth）、个人设置 |

**MVP 不做：**
- 付费系统（免费优先期不收费）
- 团队协作/多租户
- 品牌调性模板
- 批量改写
- 连续滑块式语气调节（用三档预设替代）

### Growth Features（Phase 2, 上线后 3-6 个月）

| 能力 | 说明 |
|---|---|
| 付费模式上线 | 基于 Phase 1 数据设计定价（预计：个人版 29 元/月） |
| 更多平台 | 抖音口播脚本、微博（基于用户反馈优先级排序） |
| 浏览器插件 | 在内容平台页面内一键触发改写 |
| 语气连续调节 | 从三档预设升级为连续可调滑块 |
| 平台运营建议 | 改写时附带配图方向、发布时间建议 |
| 个人风格学习 | AI 学习用户各平台写作偏好 |
| 偏好记忆 | 记住用户的语气选择，下次默认使用 |

### Vision（Phase 3, 上线后 6-12 个月）

| 能力 | 说明 |
|---|---|
| 团队版 | 多人协作、品牌调性模板、批量改写 |
| 开放 API | 供第三方工具（融媒宝/蚁小二）集成改写能力 |
| 爆文策略融合 | 输入爆文链接，AI 学习其表达策略应用到改写 |
| 长文拆解 | 一篇长文自动拆成系列短内容 |
| MCN 机构版 | 多账号管理、权限分级、批量工作流 |

## User Journeys

### Journey 1：个人博主小雨的跨平台改写（Primary - Happy Path）

**Opening Scene：** 小雨刚在小红书发了一篇 500 字的护肤体验帖，效果不错。她想把同样的内容发到公众号和知乎，但不知道怎么改。

**Rising Action：**
1. 小雨打开适文网站，将小红书帖子全文粘贴到输入框
2. 勾选"微信公众号"和"知乎"作为目标平台
3. 点击"开始改写"

**Climax：**
4. 系统先生成公众号版本（流式逐字呈现），完成后自动开始知乎版本：
   - **公众号版本：** 原来 500 字的短帖被扩展为 1200 字的深度体验文，加入了产品成分分析和使用前后对比的叙述结构，语气从"姐妹们！"变成了专业但亲切的长文风格
   - **知乎版本：** 改写为问答格式"XX 产品值得买吗？"，加入了理性分析段落和替代产品对比，语气变得客观有据
5. 每个版本附带 3 个备选标题、推荐标签和互动引导语

**Resolution：**
6. 小雨对公众号版本的第 2 个标题满意，选择"口语化"风格重新生成了一版，结果更自然
7. 复制全部内容，分别粘贴到公众号编辑器和知乎回答框，微调几句后发布
8. 点击"有帮助"反馈按钮
9. 整个过程从粘贴到发布 < 10 分钟（之前手动改写需要 1 小时+）

**情感弧线：** 好奇 -> 惊喜（"比我自己写的还好"）-> 满足 -> 依赖

**需要的能力：** 文章粘贴输入、多平台串行改写、流式输出、标题/标签生成、语气风格选择、一键复制、反馈按钮

### Journey 2：企业运营阿杰的高效工作流（Primary - Power User Path）

**Opening Scene：** 阿杰是某消费品牌新媒体运营，周一上午需要将一篇 2000 字的品牌公众号推文改写成小红书种草帖和知乎评测文。

**Rising Action：**
1. 阿杰登录适文，输入公众号原文 URL，系统尝试自动提取正文
2. URL 提取成功，阿杰确认预览内容无误
3. 勾选小红书和知乎，选择"标准"语气风格
4. 点击"开始改写"

**Climax：**
5. 两个平台版本依次流式生成：
   - **小红书版：** 精简为 300 字种草帖，结论前置，加入 emoji 和体验感表达
   - **知乎版：** 改写为 1500 字产品深度评测，加入数据对比和理性分析
6. 每个版本附带内容发布包

**Resolution：**
7. 阿杰逐一复制到各平台后台，微调品牌专属用语后排期发布
8. 改写记录自动存入历史，方便下次复用
9. 整个过程 10 分钟完成（之前需要 1.5 小时）

**情感弧线：** 压力 -> 掌控感 -> 成就感 -> 推荐给同行

**需要的能力：** URL 提取正文（Best Effort）、多平台串行改写、语气风格选择、历史记录

### Journey 3：首次用户的注册与上手（Onboarding Path）

**Opening Scene：** 新用户通过搜索"小红书文章怎么改成公众号"找到适文官网。

**Rising Action：**
1. 落地页展示产品核心价值：一篇文章 -> 多平台原生内容
2. **无需注册即可试用**：用户直接在落地页粘贴一段文章，选择一个目标平台
3. 看到改写结果预览（部分遮挡），提示"注册免费解锁完整内容"

**Climax：**
4. 用户通过手机号/微信快速注册（< 30 秒）
5. 注册后看到完整改写结果 -- "Aha" 时刻发生

**Resolution：**
6. 用户自由使用（免费期不限次数），逐渐融入日常工作流
7. 产品积累用户数据和反馈，为后续付费模式设计提供依据

**情感弧线：** 好奇 -> 试用 -> 惊喜 -> 注册 -> 常规使用

**需要的能力：** 落地页试用体验、手机号/微信快捷注册、新手引导

### Journey 4：改写结果不满意时的调整（Error Recovery Path）

**Opening Scene：** 用户完成改写后，发现某个平台的结果不够理想 -- 语气太正式或结构不太对。

**Recovery Action：**
1. 用户查看改写结果，觉得小红书版本的语气太"端着"
2. 用户切换语气风格为"口语化"，点击"重新改写"
3. 系统用新的语气设定重新生成（流式输出）
4. 如果仍不满意，用户可以直接在文本框中手动编辑改写结果
5. 用户也可以再次点击"重新改写"获取另一个全新版本

**Resolution：**
6. 经过 1-2 次调整，用户获得满意的结果
7. 用户点击"有帮助"或"没帮助"反馈

**情感弧线：** 不满 -> 调整 -> 改善 -> 满意

**需要的能力：** 语气风格切换、重新改写、结果文本可编辑、反馈按钮

**Opening Scene：** 系统运维需要监控平台健康度和成本。

1. 查看仪表盘：日活用户数、改写次数、API 调用量、API 成本、反馈满意率

5. 运维团队更新对应平台的规则配置（与 AI 改写逻辑解耦的配置层）
6. 更新后验证改写质量是否符合新规范

**需要的能力：** 管理后台仪表盘、用户行为监控、平台规则配置管理、反馈数据分析

**Rising Action：**

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. 按内容类型分策略的改写引擎**
不同于"一套规则改写所有内容"，适文首先识别原文内容类型（观点分析类 / 体验叙事类 / 教程列表类 / 评测对比类等），然后为每种内容类型 x 每个目标平台选择对应的改写策略。

例如：
- 观点分析类 -> 小红书：提取核心观点，用案例和比喻替代论证，结论前置
- 体验叙事类 -> 知乎：补充理性分析段落，加入对比数据，客观化语气
- 教程列表类 -> 公众号：整合为完整文章结构，补充背景和原理说明

**技术实现路径（MVP）：** 单次 LLM prompt 调用，将内容类型识别和改写合并在一个精心设计的 prompt 中完成。不做多步 pipeline。架构阶段评估是否需要升级为多步方案。

**2. 内容发布包（Content Package）概念**
改写输出不是单一文本，而是结构化的"发布包"：改写文案 + 备选标题 + 推荐标签 + 互动引导语。降低用户从"获得改写结果"到"实际发布"之间的摩擦。

### Market Context & Competitive Landscape

- **融媒宝/蚁小二：** 分发工具，不做深度改写。互补关系，未来可通过 API 打通
- **有一云AI/壹伴：** 排版工具，AI 是附加功能，改写质量不是核心竞争力
- **笔尖AI：** 最接近的竞品，做"爆文复刻"，但本质是伪原创（换词替换），不是语义改写
- **ChatGPT/文心一言：** 通用 AI 可以做改写，但需要用户自己写 prompt、理解平台规则。适文的价值是"把专业知识产品化为开箱即用的体验"

### Validation Approach

- **改写可用率（行为数据）：** 用户复制结果后不返回重新生成的比例 > 70%
- **用户主动反馈：** "有帮助/没帮助"按钮中正面反馈 > 70%
- **A/B 测试（Phase 2）：** 适文改写版本 vs 手动改写版本在目标平台的互动数据对比

### Risk Mitigation

- **改写质量不达标：** 按平台 x 内容类型建专属 prompt 模板 + few-shot 语料库，MVP 前大量人工质量测试
- **平台 AI 内容检测：** 改写策略侧重"重组表达"而非"换词替换"，确保原创性
- **平台规则频繁变更：** 规则配置层与 AI 逻辑解耦，支持热更新

## Technical Constraints

适文是一个 Web 应用，前端为响应式设计（支持移动端浏览器），后端 API 调用 LLM 服务完成改写。

### 认证与用户管理

- 支持手机号 + 验证码注册/登录（国内用户主流方式）
- 支持微信 OAuth 快捷登录
- 用户会话管理，支持自动续期

### 使用限制

- MVP 免费期：不限制改写次数（监控 API 成本，必要时设置合理上限）
- 全局限制：单次改写原文不超过 5000 字
- 并发限制：每用户同时最多 1 个改写任务（串行生成多平台结果）
- 反滥用：异常高频使用触发人机验证

### 关键技术约束

- 前端需支持流式文本渲染（接收后端流式响应并逐字展示）
- LLM API 调用需实现重试（1 次自动重试）、超时处理（30 秒超时）和降级方案（备选模型）
- 平台规则配置层与 AI 改写 prompt 逻辑解耦，规则变更时只需更新配置，不需要修改代码
- URL 正文提取为 Best Effort 功能 -- 提取失败时清晰引导用户手动粘贴
- 静态资源需使用 CDN 加速（面向国内用户）
- 单次改写 API 调用成本需控制在 0.3 元以内（直接关系免费期的烧钱速度）

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

采用 **免费优先 + Problem-Solving MVP** 策略：

1. **免费优先：** MVP 阶段全免费，不急于收费。目标是验证改写质量、积累用户和反馈、建立口碑。当用户基础和产品满意度达标后，再引入付费模式
2. **聚焦核心问题：** 只做 3 个平台的改写，只做一件事 -- 让内容在每个平台都像原生内容
3. **极致精简：** 砍掉一切非核心功能（团队版、连续滑块、付费系统、批量改写），把所有精力投入改写质量

### MVP Feature Set（Phase 1）

**Must Have：**
- 文章输入（粘贴全文为主 + URL 提取 Best Effort）
- 3 个目标平台改写（小红书 / 公众号 / 知乎）
- AI 改写引擎（内容类型识别 + 按平台分策略改写）
- 内容发布包输出（文案 + 标题 + 标签 + 引导语）
- 语气风格三档选择（口语化 / 标准 / 正式）
- 流式输出
- 改写反馈按钮（有帮助/没帮助）
- 历史记录（查看/复用）
- 用户注册/登录（手机号 + 微信 OAuth）
- 响应式 Web 界面
- 管理后台（基础仪表盘 + 用户管理 + 规则配置）

**Phase 1 不做：**
- 付费系统
- 团队协作 / 多租户
- 品牌调性模板
- 批量改写
- 连续语气滑块
- 偏好记忆

### Post-MVP Features

**Phase 2 - Growth & Monetization（上线后 3-6 个月）：**
- 付费模式上线（基于用户数据设计定价）
- 新增平台：抖音口播脚本、微博（按用户需求排序）
- 浏览器插件（Chrome/Edge）
- 语气连续调节器（滑块）
- 偏好记忆（记住用户语气选择）
- 个人风格学习
- 平台运营建议

**Phase 3 - Expansion（上线后 6-12 个月）：**
- 团队版（多人协作、品牌调性模板、批量改写）
- 开放 API
- 更多平台支持
- 长文拆解为系列短内容
- 爆文策略融合改写
- MCN 机构版

### Risk Mitigation Strategy

| 风险 | 影响 | 概率 | 缓解措施 |
|---|---|---|---|
| AI 改写质量不达标 | 致命 | 中 | 按平台 x 内容类型建专属 prompt 链 + few-shot 语料库，MVP 前大量人工质量测试 |
| 免费期 API 成本失控 | 致命 | 中 | 单次成本控制在 0.3 元内，选最优性价比模型，优化 prompt 长度，异常使用触发人机验证 |
| URL 正文提取被平台反爬 | 中 | 高 | 标记为 Best Effort，失败时清晰引导粘贴，不将 URL 提取作为核心依赖 |
| 平台规则快速变化 | 中 | 高 | 规则配置层与 AI 逻辑解耦，支持热更新 |
| 用户获取成本高 | 中 | 中 | 免费策略降低获客门槛 + 内容 SEO + 口碑传播 |
| 竞品跟进（分发工具加 AI） | 中 | 高 | 专注改写质量深度，他们做 60 分改写我们做 90 分 |
| 免费期结束后付费转化率低 | 中 | 中 | 免费期积累足够用户反馈和行为数据，数据驱动定价决策 |

## Functional Requirements

### Content Input

- **FR1:** 用户可粘贴纯文本内容作为改写原文（支持 50-5000 字）
- **FR2:** 用户可输入 URL，系统尝试自动提取文章正文内容；提取失败时显示明确提示并引导用户手动粘贴
- **FR3:** 系统在输入后显示原文预览，供用户确认内容正确
- **FR4:** 系统对超出字数限制或低于最低字数的内容给出明确提示

### Content Type Detection

- **FR5:** 系统在改写前自动识别原文内容类型（观点分析类 / 体验叙事类 / 教程列表类 / 评测对比类 / 其他）
- **FR6:** 系统按识别出的内容类型选择对应的改写策略
- **FR7:** 对于无法有效改写的内容（如纯代码、纯数据表格、外语内容），系统给出明确提示

### Platform Selection & Configuration

- **FR8:** 用户可选择一个或多个目标平台进行改写（小红书 / 微信公众号 / 知乎）
- **FR9:** 用户可选择语气风格预设（口语化 / 标准 / 正式三档）
- **FR10:** 用户选择新的语气风格后，可点击"重新改写"以新风格重新生成

### AI Rewriting Engine

- **FR11:** 系统按目标平台的内容结构规范重组内容（小红书：体验+结论前置；公众号：论点递进+深度分析；知乎：问题拆解+数据支撑）
- **FR12:** 系统用目标平台的语言风格重写表达（用词、句式、emoji 使用规范、互动口吻）
- **FR13:** 多个目标平台的改写串行生成（完成一个再开始下一个），每个平台以流式方式逐字呈现
- **FR14:** 用户可在改写完成后点击"重新改写"获取全新版本

### Content Package Output

- **FR15:** 每个目标平台的改写结果包含：改写文案全文 + 3 个备选标题 + 推荐标签（3-5 个）+ 互动引导语
- **FR16:** 用户可一键复制改写结果中的任意组成部分（文案 / 标题 / 标签 / 引导语）
- **FR17:** 用户可直接在结果区域编辑改写内容
- **FR18:** 多个目标平台的改写结果在同一页面内以 tab 或卡片形式展示，默认显示文案，标题/标签/引导语可折叠展示

### Feedback

- **FR19:** 每次改写结果旁展示"有帮助/没帮助"反馈按钮
- **FR20:** 用户点击"没帮助"时，可选填简短文字说明原因

### History & Content Management

- **FR21:** 系统自动保存每次改写记录（原文 + 所有平台改写结果 + 使用的配置）
- **FR22:** 用户可查看改写历史列表，按时间倒序排列
- **FR23:** 用户可从历史记录中重新加载原文并发起新的改写

### User Authentication & Account

- **FR24:** 用户可通过手机号 + 短信验证码注册和登录
- **FR25:** 用户可通过微信 OAuth 快捷注册和登录
- **FR26:** 系统维护用户会话状态，支持自动续期
- **FR27:** 用户可在个人设置中查看和修改基本信息

### Admin & Operations

- **FR28:** 管理后台展示系统运行仪表盘（DAU、改写次数、API 调用量、API 成本、反馈满意率）
- **FR29:** 管理后台可查看和管理用户账号
- **FR30:** 管理后台可更新平台规则配置（与 AI 改写逻辑解耦）
- **FR31:** 系统记录每次改写的完整日志（原文类型、目标平台、API 耗时、成本、用户反馈）

## Non-Functional Requirements

### Performance

- NFR1: 系统在用户点击"开始改写"后 2 秒内开始流式输出第一个 token
- NFR2: 2000 字以内原文的单平台完整改写在 15 秒内完成
- NFR3: 页面首次加载时间（LCP）< 2 秒（国内 CDN 加速）

### Security

- NFR4: 用户手机号、OAuth token 等认证信息使用加密存储
- NFR5: API 通信全程使用 HTTPS
- NFR6: 用户改写内容在不同账户之间完全隔离，不可交叉访问
- NFR7: LLM API Key 等敏感配置不暴露给前端

### Scalability

- NFR8: 系统架构支持从 100 并发用户平滑扩展到 10,000 并发用户
- NFR9: LLM API 调用层支持多提供商切换（主提供商不可用时降级到备选）

### Reliability

- NFR10: 系统月度可用性 > 99.5%
- NFR11: LLM API 调用失败时自动重试一次，仍失败则向用户展示明确错误信息和重试按钮
- NFR12: 用户改写历史数据定期自动备份

### Integration

- NFR13: URL 正文提取支持微信公众号、知乎、小红书文章链接（Best Effort，允许失败并优雅降级）
